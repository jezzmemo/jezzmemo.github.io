如何监控移动产品

最近在思考让产品稳定的向前发展，发现我们在产品的监控环境做的不是很好，仅仅只是嵌入了一个SDK，虽然有一些运营数据和Bug收集功能，
但是对产品的业务没有检测，不能很好让开发和产品运营相互配合，不利于产品更好的完善以及用户体验，对于发布出去的App要有很好的掌控，
在参考了各个大公司的监测手段后，我对移动产品的监测总结如下.

热修复方案

  • JSPatch
  • LuaView
  • React Native

以上三种方案，是我听的比较多，而且网络上讨论的比较多的三种方案:

  • JSPatch是现在比较流行的，因为学习简单，接入成本低，功能强大,所以在很多App里都有接入，技术原理是充分利用了OC的Runtime特性
    和JS之间的配合
  • 聚划算iOS平台动态化方案LuaView,实现是通过Lua为主的脚本语言，特点就是iOS和Andoird平台是同一套代码
  • JSPatch在学习成本，接入成本，热更新能力上占优，而 React Native 在开发效率和跨平台能力上占优，大家可以根据需求的不同选用不同的热更新方案

连续闪退自我修复

首先分析什么情况会出现反复闪退

  • 数据库损坏
  • 文件损坏
  • 网络数据异常
  • 代码bug

通过计数检测App有没反复闪退，如果有闪退通过步骤恢复：
1.修复数据库
2.删除本地所有数据(如果远程有数据，可以直接删除，如果没有先要备份到网络上，以免丢失用户有价值数据)
3.检测有没下发的Patch更新，如果没有，将上次Crash原因上报到系统
4.重新进入初始化过程

Crash报错

  • Mach 异常：EXC_CRASH
  • UNIX 信号：SIGABRT
  • NSException

iOS的错误处理机制不在本地的讲解中，如果有兴趣可以到念茜的漫谈 iOS Crash 收集框架

我们将所有的错误列处理，如果出现错误了，我们将错误上传到服务器。

性能监测

  • 卡顿监测(Runloop机制，监测每次主线程的执行时间)
  • 页面渲染时间和请求时长
  • CPU,内存，FPS，网络负载

卡顿监测和页面的请求时间是很重要的，因为这很影响用户的体验，而且是很直接的，所以这部分数据是很有价值的.

CPU:分析那块业务复杂程度，为什么会长时间很高，带来的直接问题就是用户响应慢，耗电量快
内存:主要用来分析泄露以及不恰当的内存分配方式
FPS:用来反映的流畅程度
网络负载:反映App的网络使用情况，不论是bug，还是流量监控

业务监测

  • 数据异常
    如:书本打不开，下载经常失败

  • 功能异常
    如:没有按照自己既定的逻辑运行

自动化测试监测

根据自己的业务，定制制动化测试计划，让软件一直处于运行状态，从而监测软件的运行健壮性和业务问题,
这样能比较全面的覆盖到，无论是App的bug,还是测试业务问题，以及架构的优缺点，对于公司内部的自我升级是很有帮助的，
所以自动化测试环节是必不可少的.

发送监控数据形式

  • 短信提醒
  • 邮件提醒
  • 自动电话
  • 自动IM提醒

所有的问题都会指定到人，问题定义严重程度，来决定用短信或邮件发送，但是邮件是所有问题的必走渠道.
还有一个问题就是，我们的监控数据或者需要优化的数据怎么让开发重视起来，不能简单的发送过去，我们要让开发重视起来并去解决这些问题，才能让整个监控的流程形成一个闭环，这样才是良性循环.

用户反馈

  • 软件内反馈
  • 热心网友电话反馈

用户反馈，虽然以上两种是效率很低的，在我们技术有遗漏的时候，这种原始的方式还是很有效的，这种让我们监控移动产品比较全面，让产品正向的发展很重要的手段

以上是我最近对产品检测的思考和总结，我首先是整体的规划下，以上每个话题都是可以单独开一篇来详细讨论，
所以不排除在接下来的时间来一一完善我们文章里提到的重点和解决方案,让整个监控真实反映出我们各个环境可能存在的问题，这样才能把产品推到一个比较高的高度.
