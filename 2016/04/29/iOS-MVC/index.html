iOS之MVC

最近一直在关注架构这块的内容以及思考，首先我们要明确一个问题，我们为什么要关注架构，架构给我们带来的好处?
以下回答都是基于我个人的理解，架构的产生就是为了把复杂问题简单化，在多个人协作的时候高效率的运作，以及后期维护的易用性和可靠性，假如没有良好的架构设计，我们会在一个复杂的环境中工作,各种问题就凸显出来，Bug的几率就很高，后期的维护工作就会变得很差。

provides a high-level MVC architecture and automates communication between models,views, and controllers.

为什么我这次专门讨论MVC架构，因为的这种架构很简单，很容易理解和接受，让大多数的开发人员快速的上手并应用到实际的项目当中,还有一种说法就是,MVC是一种抽象级别很高的，其他的MVP,MVVM等等其他的架构方式，都是基于MVC这种模式演变过来的，所以MVC很值得我们去研究和学习.

传统MVC

根据以上图示，MVC三者之间都有通信，他们之间的藕合度很高，所以这种模式不被iOS所推荐

Cocoa的MVC

Controller是Model和View之间的协作作用，Model和View之间是不通讯的，他们两之间的协作只能通过Controller来作为中间者来通讯，这样的好处是View和Model是独立的，他们之间的复用度比较高，以及测试性也比较强.

实际的Cocoa的MVC

根据实际情况里的iOS里的MVC,View和Controller之间是耦合的，这样带来的问题就是Controller负责的任务比较多，比如：

  • 网络请求
  • UI 的调整和组合
  • 业务逻辑
  • Delegate、Data source

所以Massive View Controller就是这样出现的，基于这种情况，测试性其实只有Model是独立可测试性的，View和Controller是耦合的，可测试性很差.

总结

根据以上情况我们可以总结，iOS的MVC就开发速度来说是一个不错的选择方案，因为学习成本低，团队都能容易接受这种架构，比如你的团队在磨合阶段选择这个也是一个不错的选择，不过问题也很明显，就是Controller的臃肿，工程在不断的庞大过程中表现的很突出，所以大家要根据大家的实际情况来选择最适合自己的方案。
